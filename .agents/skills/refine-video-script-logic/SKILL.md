# 视频文案逻辑初审

在导演分镜和Seedance编译之前，对用户原稿做一次轻量、可追溯、必须确认的逻辑初审。

## 职责边界

- 只审查和提出候选调整，不直接制作分镜、资产或Seedance提示词。
- 保留原稿的人物、关系、核心梗、结局和用户指定台词。
- 优先做最小修改，不把用户文案改造成另一条故事。
- 区分“逻辑必须修复”“传播或节奏建议”“个人审美建议”。
- 用户未确认前，所有修改稿均为候选，原稿仍是唯一锁定版本。
- 如果用户要求从零想三个方向，转交 `create-video-concept`。
- 如果用户要求大幅重写完整剧本，转交 `build-video-script-storyboard`。
- 用户锁定文案后，才交给 `compile-seedance2-shots`。
- 对抖音情绪短片、命题短片、参考爆款迁移、答案型/群像文案、产品故事、亲情/友情/遗憾/人生哲理类短片，逻辑初审必须同时做爆款前置审查：强钩子、用户停留、情绪递进、说话关系、精简深意台词和结尾回扣。若缺少这些基础，不得只做轻微润色后放行。

## 前置读取

只读取本次需要的资料：品牌定位、目标用户、对应产品卡、产品内容总原则，以及用户明确指定的少量训练规则。产品卡为V0时，不补写功效、配料或未经确认的卖点。

## 工作流程

### 1. 锁定原稿

编号并原样记录：环境、人物、关系、目标、关键动作、对白、信息揭示、产品作用、反转和结尾。列出用户明确要求“必须保留”和“允许调整”的内容。

### 2. 检查逻辑

读取 `references/review-checklist.md`，逐项检查：

- 人物为什么这样做，是否符合身份和利益。
- 前因能否自然引出后果。
- 关键物品、金钱、手机或产品是否有合理用途。
- 观众在每个阶段知道什么，反转前是否提前泄底。
- 反转揭晓后，前文能否重新解释且没有硬伤。
- 最后的笑点、情绪或观点是否来自故事，而不是强喊口号。

### 2.5 爆款叙事验收

若文案属于命题短片、情绪短片、参考爆款迁移、群像答案型文案、亲情、友情、遗憾、成长或错过主题，进入下游前必须检查：

- 前 3-8 秒是否有停留钩子。
- 前 20 秒是否给出继续观看理由。
- 中段是否至少有 3 次情绪升级，而不是并列清单。
- 每段是否有角色、此刻愿望、困难、选择和余味。
- 最后三段是否明显重于前三段。
- 结尾是否回扣开头的具体动作、物件、台词、问题或关系。
- 主题是否由故事推出，而不是旁白宣布。
- 是否有明确情绪阶梯：轻冲突 -> 现实处境 -> 关系代价 -> 不可逆遗憾 -> 主题回扣。若段落顺序可随意调换，视为未通过。
- 关键台词是否精简、有对象、有未说出口的代价，并让观众从处境中悟出哲理；不得由角色直接喊出抽象人生道理。

未通过时，输出“退回 G0方向/开场重构或G1文案重构”的建议，不得建议进入导演分镜或 Seedance 编译。

### 2.5.1 8秒开场复核

若文案开头弱，必须提供至少三种开场替代候选：

1. 认知错位型。
2. 轻问题重答案型。
3. 反常动作/关系悬念型。

每个候选只写前3-8秒，不直接重写全文；比较停留理由、信息密度、情绪承诺和Seedance可生成风险。用户确认开场方向后，再继续全文重构。

### 2.6 答案型文案说话关系检查

如果文案采用街采、群像回答、多人依次回答同一问题、答案池或金句型结构，必须逐条检查：

- 角色身份是否具体。
- 这句话真正说给谁听。
- 关键对象是否缺席；若在场，为什么仍说不出口。
- 这句话没说、没做或没等，会留下什么代价。
- 画面中有没有可见锚点承载情绪。
- 哲理是否由处境推出，而不是角色直接讲道理。
- 入片答案是否足够短，是否存在可以删掉的解释句、总结句或作者替角色说出的哲理。

未通过时，不能进入分镜；应退回补写说话关系表。

### 2.7 角色台词精简深意检查

对所有关键台词逐句检查：

- 这句话真正说给谁。
- 为什么此刻难说。
- 没说、晚说或说错会留下什么代价。
- 画面里由哪个动作、物件、空间或停顿承接。
- 隐含的人生哲理是什么。
- 台词是否可以更短，是否删掉解释后更有余味。

修改原则：

- 优先删解释，不删关系。
- 优先保留半句话、停顿、具体称呼、物件和动作。
- 角色不得直接说“人生、亲情、遗憾、成长、珍惜”等抽象结论，除非这是人物身份和场景明确需要的真实表达。
- 能用画面承载的情绪，不用台词重复说明。

### 3. 检查品牌与产品

- 产品是否自然参与人物关系或情节因果。
- 产品名称、价格、使用动作是否可能造成错误理解。
- 是否出现功效、疾病、绝对化、健康焦虑或药品化暗示。
- 删除产品后故事是否仍成立；加入产品后是否更自然，而非插播广告。

### 4. 检查Seedance可制作性

只做脚本层预判，不编译镜头。标记人物数量、场景变化、长对白、多人同框说话、精确手机文字、车辆商标、产品包装、复杂手部动作和大幅运镜等风险。

风险不等于删除。先给出拆分、简化或交给后期实现的候选方案。

### 5. 输出修改候选

按 `references/output-and-confirmation.md` 输出：

1. 原稿核心锁定。
2. 总体判断。
3. 问题清单及证据。
4. 最小修改建议。
5. A版原稿保留版。
6. B版逻辑微调版。
7. 必要时提供C版生成友好版，但不得改变核心故事。
8. 爆款叙事修复建议：开场、情绪阶梯、结尾回扣和精简深意台词。
9. 变更对照表。
10. 用户确认选项。

只对确有问题的片段提供改写，不为了展示能力而全文改写。

## 问题等级

- P0 阻塞：人物身份矛盾、事件无法成立、关键对白归属不明、产品用途完全不明却安排具体使用动作。修复前不能进入编译。
- P1 重要：反转泄底、因果牵强、产品造成明显误解、对白时长严重超出目标、合规风险。建议确认修改。
- P2 优化：节奏、措辞、笑点停顿、信息顺序或生成复杂度可以更好。用户可以保留原稿。
- P3 偏好：纯审美选择，不得描述成错误。

## 确认规则

结尾必须明确等待用户选择：

- A：保留原稿，只记录风险。
- B：采用逻辑微调版。
- C：采用生成友好版。
- D：按用户指定条目组合修改。
- E：退回继续讨论，暂不锁定。

用户确认后，输出一次“锁定文案”，标明采用版本、保留内容、修改内容、未采用建议和日期。没有确认时不得称为最终版。

## 交接条件

只有用户明确说“确认”“锁定”“采用A/B/C/D”后，才能建议调用 `compile-seedance2-shots`。交接时传递锁定文案，而不是整段审查讨论。
