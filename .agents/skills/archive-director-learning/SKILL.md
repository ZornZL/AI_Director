# 编导训练归档

## 触发条件

只有用户明确说“满意”“值得训练”“保存为训练样本”或选择入库类型时使用。

## 归档前总结

使用 `04-模板库/训练归档模板.md`，提炼用户喜欢、不喜欢、训练目标、结构、钩子、迁移场景、原创边界和标签。

## 入库选项

- A：案例、结构、钩子、偏好和最终成果完整入库。
- B：只保存案例。
- C：保存为新结构模型。
- D：暂不入库。
- M：按已展示的合并方案更新已有案例、结构、钩子、偏好或避坑卡。

用户选择前不得写入正式库。

## 执行前复核

1. 读取分析 Skill 输出的历史库对照、推荐项和合并目标。
2. 再次确认目标文件确实表达同一机制，不能只凭名称相似合并。
3. 检查用户确认的是 A/B/C/D 还是具体合并方案；不得将“推荐”当作“批准”。
4. 若用户选择 A 但已有知识高度重复，完整保存新案例，同时优先扩展已有结构、钩子和偏好卡，避免创建同义卡。
5. 若用户选择 C 但结构只是已有模型的变体，先向用户展示“新建”与“合并扩展”的差异，再执行其最终选择。

## 去重与合并规则

- 相同结构名称不同：保留更清晰的主卡名，把新案例加入来源和验证记录。
- 相同钩子仅台词不同：追加为变体，不新建钩子卡。
- 相同偏好新增场景：扩展适用场景与确认来源，不重复创建偏好卡。
- 新结论修正旧规则：保留修订记录、证据和确认日期，不静默覆盖。
- 两条规则互相冲突但用途不同：分别保留并标注适用场景、禁用场景和优先条件。
- 正向案例、未验证想法和失败经验继续分库存放。

## 写入规则

1. 完整分析放案例库。
2. 可复用公式放结构库。
3. 开场机制放钩子库。
4. 用户反馈放审美偏好或避坑库。
5. 更新训练样本登记表和能力地图。
6. 初始为L1；原创迁移满意后为L2；数据验证后为L3。
7. 合并时更新来源案例、验证次数、适用边界和版本号；不要虚增独立结构数量。
8. D 只更新项目内状态为暂不入库，不增加正向案例数，不升级能力地图。
